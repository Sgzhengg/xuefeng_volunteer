# P1任务：官方位次数据采集方案

**目标**：从5个高考大省的教育考试院采集官方专业录取位次数据，替代基于分数转换的估算数据。

3. 具有代表性（覆盖不同地区）

## 2. 数据采集策略

### 2.1 数据源分析

#### 江苏省教育考试院 (http://www.jseea.cn/)
**可采集数据**：
- 普通类本科批次录取分数和位次
- 分院校分专业录取数据
- 历年数据（2020-2024）

**采集路径**：
- 首页 > 招考信息 > 普通高校招生 > 录取分数线
- 数据格式：HTML表格 / PDF

**技术方案**：
```python
# 伪代码
url = "http://www.jseea.cn/front-end/"
data_type = "ADMISSION_SCORE_BY_MAJOR"

```

#### 浙江省教育考试院 (http://www.zjzs.net/)
**可采集数据**：
- 分院校分专业录取数据
- 普通类平行录取投档分数和位次

**采集路径**：
- 首页 > 信息公开 > 招考信息 > 高校招生 > 录取分数线

#### 河南省/山东省/广东省
**类似采集策略**，根据网站结构调整爬虫

---

### 2.2 技术实现方案

#### 方案A：传统爬虫（requests + BeautifulSoup）
**优点**：
- 简单直接
- 适合静态HTML页面

**缺点**：
- 无法处理JavaScript渲染
- 容易被反爬虫机制阻止

**适用场景**：
- 数据以静态HTML形式展示
- 网站反爬虫较弱

#### 方案B：Selenium浏览器自动化
**优点**：
- 可处理JavaScript渲染
- 模拟真实用户行为

**缺点**：
- 速度较慢
- 资源消耗大

**适用场景**：
- 数据需要JavaScript渲染
- 网站有反爬虫机制

#### 方案C：API接口调用
**优点**：
- 速度快
- 数据格式规范

**缺点**：
- 需要逆向分析API
- API可能需要认证

**适用场景**：
- 网站提供API接口
- 数据可通过AJAX请求获取

**推荐方案**：方案A + 方案C混合
- 优先尝试直接调用API接口
- 如果API不可用，使用BeautifulSoup解析HTML
- 特殊情况下使用Selenium

## 3. 数据标准化流程

### 3.1 数据清洗

**原始数据示例**（HTML表格）：
```html
<table>
  <tr>
    <td>北京大学</td>
    <td>计算机科学与技术</td>
    <td>676</td>
    <td>1500</td>
  </tr>
</table>
```

**清洗后的数据**：
```json
{
  "university_name": "北京大学",
  "major_name": "计算机科学与技术",
  "min_score": 676,
  "min_rank": 1500,
  "province": "江苏",
  "year": 2024
}
```

### 3.2 数据验证

**验证规则**：
1. 分数合理性：400 <= 分数 <= 750
2. 位次合理性：位次 > 0
3. 院校名称在标准库中存在
4. 专业代码符合标准格式（6位数字）

**异常数据处理**：
- 记录异常数据到日志文件
- 人工审核异常数据
- 修正或丢弃无效数据

### 3.3 数据集成

**目标格式**：
```json
{
  "university_major_id": "1_080901",
  "university_id": "1",
  "university_name": "北京大学",
  "major_code": "080901",
  "major_name": "计算机科学与技术",
  "province": "江苏",
  "year": 2024,
  "min_score": 676,
  "avg_score": 681,
  "min_rank": 1500,
  "avg_rank": 1200,
  "data_source": "official_jiangsu_2024",
  "accuracy": "high"  // 官方数据，高精度
}
```

**集成步骤**：
1. 解析院校名称，匹配university_id
2. 解析专业名称，匹配major_code
3. 生成university_major_id
4. 合并到major_rank_data.json
5. 标记data_source为官方来源

## 5. 风险评估

### 风险1：网站结构变化
**概率**：中等
**影响**：高
**应对**：
- 采集前先分析网站结构
- 编写灵活的解析器
- 准备多个采集方案

### 风险2：反爬虫机制
**概率**：高
**影响**：中等
**应对**：
- 使用请求头伪装
- 限制请求频率
- 使用代理IP池
- 模拟真实用户行为

### 风险3：数据格式不一致
**概率**：高
**影响**：中等
**应对**：
- 为每个省份定制解析器
- 编写通用的数据清洗函数
- 人工审核异常数据

### 风险4：数据不可用
**概率**：低
**影响**：高
**应对**：
- 优先选择数据公开的省份
- 准备备用省份列表
- 如果数据不可用，使用估算数据
